Risks and Considerations

Understanding trade-offs before adopting rg.forms

rg.forms offers significant productivity gains, but like any technology choice, comes with trade-offs. This page documents known risks and our mitigation strategies.

High Risk

Datastar Dependency

Risk: Datastar is v1.0-RC (not stable 1.0 yet) with a smaller community compared to HTMX (~25K stars) or Alpine (~29K stars). If Datastar API changes significantly or project becomes unmaintained, rg.forms is tightly coupled.

Mitigation:

  • Expression evaluator is backend-only (Python) - not Datastar-dependent
  • Frontend attributes (data-show, data-bind) could be adapted to HTMX + Alpine
  • Monitor Datastar project health and community growth
  • Core value (single source of truth) is framework-agnostic

API Stability

Risk: Field attributes have grown significantly and more will likely be needed as real-world usage reveals gaps:

  • Current: visible_when, required_when, computed, disabled_when, read_only_when, help_text_when, depends_on, choices_from...
  • Potential: error_message_when, css_class_when, choices_when

Mitigation:

  • Semantic versioning with clear deprecation policy
  • Collect real-world usage feedback before stabilizing API
  • Breaking changes only in major versions with migration guides

Expression Language Limitations

Risk: Current expression parser supports only basic operations. Users may hit limits:

MissingExample
Functionslen($items), $date.year
Ternary$a ? $b : $c
Array access$items[0], $items.length
String opsstartsWith, contains
Nested fields$address.city

Mitigation:

  • Document supported expression subset clearly
  • Escape hatch: custom clean_* methods for complex logic
  • Expand parser based on most-requested features

Medium Risk

Missing Django Integrations

Risk: Common Django patterns not yet supported:

  • ReactiveModelForm - most common use case
  • Formsets / InlineFormsets
  • FileField with upload progress
  • crispy-forms integration

Mitigation: Prioritize ReactiveModelForm for next phase. Document workarounds.

Expression Errors are Silent

Risk: Typos in expressions ($ordertpye instead of $order_type) fail silently, defaulting to visible. Debugging is difficult.

Mitigation:

  • Add DEBUG mode that raises on expression errors
  • Add form validation at definition time
  • Provide check_expressions() method for testing

No IDE Support for Expressions

Risk: Expression strings like "$order_type == 'urgent'" have no autocomplete, type checking, or refactoring support.

Mitigation: Document syntax clearly. Consider future TypedDict approach. Provide testing utilities.

Lower Risk

Single CSS Framework

Only Bulma templates. Bootstrap 5 planned. Template override mechanism available.

Performance at Scale

Server round-trip for cascading. Mitigated by Datastar's debouncing and fast expression parsing.

No Async Validation

No built-in "check availability" pattern. Custom clean_* methods can call external services.

When NOT to Use rg.forms

Consider alternatives if you need:

  • Offline-first - Forms must work without network connectivity
  • Complex client-side state - Multi-step wizards with heavy local manipulation
  • React Native code sharing - Same form logic on web and mobile apps
  • Existing SPA investment - Team already proficient in React/Vue ecosystem
