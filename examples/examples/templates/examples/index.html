Usage Pattern

Auto-generated

{% render_reactive_form form %}

Use for standard vertical forms where field order matches form definition.

Manual layout

<form data-signals='{% reactive_signals form %}'>
  {% render_reactive_field form.field1 %}
  ...
</form>

Use for custom layouts, multi-column, or special UI requirements.

Why rg.forms? Development Effort Comparison

Building reactive forms with traditional SPA stack vs. rg.forms:

Aspect React + Formik + DRF rg.forms + Datastar
Codebase size ~3x more code (frontend + backend + API) Single Python form class
Validation logic Duplicated: Yup/Zod (JS) + serializers (Python) Single source of truth in Python
Build pipeline Node.js, npm, webpack/vite, ~40s builds None needed
Dependencies 200+ npm packages typical Django + Datastar (14KB)
Security surface 200+ packages = 200+ potential vulnerabilities Minimal attack surface
DevOps / CI npm install, webpack build, separate Docker layers Standard Django deployment
API layer REST endpoints, serializers, CORS, auth tokens Standard Django form POST
Data transfer JSON API often larger (includes unused fields) HTML contains only what's displayed
First page load Blank → JS download → parse → hydrate → render HTML renders immediately
Client CPU/memory Virtual DOM diffing, state management overhead 10-100x less CPU (native DOM, no framework)
Team structure Separate FE/BE developers required Full-stack Django developers
FE/BE coordination API contracts, schema sync, meetings Not needed - single codebase
Onboarding Weeks (JSX, hooks, state, Virtual DOM) Days (Django forms + HTML attributes)
Debugging Browser DevTools + network + React DevTools Django debug toolbar + view source

Real-world case study: A company replaced 2 years of React work with HTMX/templates in 2 months, achieving 67% code reduction (21K → 7K LOC) and 96% fewer JS dependencies.

Source: htmx.org - A Real World React to htmx Port

Estimated Development Time

For a typical form with conditional visibility, dynamic requirements, computed fields, and cascading dropdowns:

React + Formik + DRF

  • API design & documentation: 2-4h
  • Backend serializers & views: 4-8h
  • Frontend form components: 8-16h
  • Validation sync (FE ↔ BE): 2-4h
  • FE/BE developer coordination: 2-4h
  • Testing both sides: 4-8h
  • Total: 22-44 hours

rg.forms + Datastar

  • Form class definition: 1-2h
  • View (standard Django): 0.5-1h
  • Template: 0.5-1h
  • Testing: 1-2h
  • Total: 3-6 hours

~5-7x faster development

When to use SPA instead?

rg.forms is ideal for 90% of business forms. Consider React/Vue when you need:

  • Offline-first with local state persistence
  • Heavy client-side data visualization (D3.js, Three.js)
  • Mobile app with React Native code sharing

Note: Real-time collaboration works great with Datastar + SSE. See example.
